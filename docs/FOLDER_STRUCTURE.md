# Folder Structure

This document outlines the organization of the Jarvis AI Assistant project.

## Root Folders

| Folder | Purpose | Examples of Files | Why it Exists |
| :--- | :--- | :--- | :--- |
| **`app/`** | Contains the core Python source code and packages. | Python modules, packages, UI definitions. | Standardizes Python package layout and encapsulates logic. |
| **`assets/`** | Stores static media resources used by the UI and system. | Icons, logos, UI images, sound effects. | Centralizes assets for easy deployment packaging. |
| **`docs/`** | Houses design specs, user documentation, and guides. | `ARCHITECTURE.md`, `ROADMAP.md`. | Keeps project details, plans, and technical documentation organized. |
| **`tests/`** | Contains unit, integration, and end-to-end tests. | `test_agent.py`, `test_memory.py`. | Encourages test-driven development and prevents regressions. |
| **`scripts/`** | Houses administration and development scripts. | Build scripts, database migration scripts. | Automates setup, build, and run tasks for developers. |
| **`logs/`** | Stores runtime log files generated by the application. | `app.log`, `error.log`. | Aids debugging and diagnosing runtime issues in production. |
| **`data/`** | Holds local application data and database files. | `jarvis.db`, cached model metadata. | Keeps user data separate from code and configuration. |

---

## Application Packages (`app/`)

The core application code is divided into the following modules under `app/`:

### 1. `ai`
- **Purpose**: Manages communication with LLM APIs and local inference engines.
- **Why it exists**: Isolates AI model integrations from other business logic.

### 2. `core`
- **Purpose**: Houses the agent loop, state machine, and orchestrator code.
- **Why it exists**: Serves as the nervous system coordinating overall system actions.

### 3. `config`
- **Purpose**: Handles parsing of environment variables, YAML config, and runtime options.
- **Why it exists**: Provides a unified, type-safe configuration access point across the system.

### 4. `memory`
- **Purpose**: Implements short-term conversation storage, long-term memory, and SQLite database schemas.
- **Why it exists**: Ensures the agent maintains context across messages and restarts.

### 5. `prompts`
- **Purpose**: Contains system prompt templates, task prompts, and dynamic prompt builders.
- **Why it exists**: Decouples the prompt engineering lifecycle from core application code.

### 6. `services`
- **Purpose**: Houses external service integration clients (e.g., calendar, email).
- **Why it exists**: Modularizes connections to third-party APIs.

### 7. `tools`
- **Purpose**: Defines system tools, runtimes, sandboxing, and execution schemas.
- **Why it exists**: Allows safe execution of functions requested by the AI.

### 8. `ui`
- **Purpose**: Implements the desktop user interface windows, dialogs, and styles.
- **Why it exists**: Separates presentation layer views from underlying logic.

### 9. `utils`
- **Purpose**: Contains global helper functions, date/time formatting, and file helpers.
- **Why it exists**: Prevents code duplication of common utility operations.

### 10. `voice`
- **Purpose**: Handles text-to-speech (TTS) and speech-to-text (STT) processing.
- **Why it exists**: Abstracts voice capabilities to maintain conversational agility.
